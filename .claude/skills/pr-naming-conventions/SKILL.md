# PR Naming Conventions for Expo + EAS Update Project

## Overview

This project uses Expo Dev Client + EAS Update. PR labeling determines the deployment method, so proper naming and labeling is critical.

## Branch Naming Convention

When creating new branches, use the following format:

### Format
```
<type>/<description>
```

### Types
- `feature/` - New features
- `fix/` - Bug fixes
- `refactor/` - Code refactoring
- `docs/` - Documentation updates
- `test/` - Test additions
- `chore/` - Maintenance tasks
- `claude/` - Changes made by Claude Code (auto-generated)

### Examples
- `feature/eas-update-workflow`
- `fix/env-variable-issue`
- `docs/setup-instructions`
- `claude/setup-gh-actions-cA5rE` (auto-generated by Claude Code)

## PR Title Format

PR titles should be clear and concise:

```
[Type] Brief description of changes
```

### Examples
- `[Feature] Add EAS update workflow`
- `[Fix] Resolve environment variable in commit message`
- `[Docs] Update setup guide`
- `[Chore] Update dependencies`

## PR Labeling Rules (CRITICAL)

**This determines the GitHub Actions workflow that runs on merge:**

### No Label (Default)
- **For:** JS-only changes (UI, business logic, API calls, text/strings)
- **Workflow:** `eas update` runs on merge (fast, free)
- **Examples:**
  - UI component changes
  - Screen updates
  - Business logic modifications
  - API call changes
  - Text/string updates

### `native` Label
- **For:** Native code changes requiring rebuild
- **Workflow:** `eas build` runs on merge (slow, counts against quota)
- **Examples:**
  - Intent handlers / deep links
  - New permissions
  - Native modules
  - Package name changes
  - App icon updates
